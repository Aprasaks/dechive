# Dechive.dev 디자인/UX 방향 정리

## 1. 핵심 포지셔닝

Dechive는 단순한 블로그가 아니라 **지식의 도서관**으로 가야 한다.

한 줄 정의 후보:

> Dechive — 지식을 탐색하는 도서관

또는

> 생각이 기록이 되는 순간, 의미를 가진다.

중요한 건 문구 자체보다, 사용자가 들어오자마자 이렇게 느끼게 만드는 것이다.

> “여기 글 목록이 아니라, 지식이 쌓여 있는 공간이네.”

---

## 2. 지금까지의 큰 판단

### 유지해야 할 것

- 고딕 도서관 분위기
- 어두운 배경과 따뜻한 촛불 톤
- 책장/아카이브/기록 감성
- 해골 마스코트
- Archive의 펼쳐진 책 UI

### 갈아엎지 않아도 되는 것

- 전체적인 브랜드 톤
- Archive의 좌우 책 구조
- 다크 판타지 도서관 배경 방향

### 바꿔야 할 것

- “예쁜 랜딩”에서 “탐색 가능한 도서관”으로 전환
- 단순 글 목록이 아니라 탐색 루프 만들기
- 마스코트를 장식이 아니라 사서 인터페이스로 사용

---

## 3. Dechive의 핵심 경험

가장 중요한 문장:

> 도서관은 사용자가 길을 잃을 수 있어야 완성된다.

단, 여기서 “길을 잃는다”는 뜻은 무작위로 방치한다는 의미가 아니다.

> 의미 있는 방황을 설계한다는 뜻이다.

Dechive가 만들어야 할 경험:

```txt
검색/탐색 → 글 읽기 → 연관 글 → 다른 시리즈 → 랜덤 발견 → 다시 탐색
```

즉, 사용자가 한 글만 읽고 끝나는 것이 아니라, 계속 움직이게 해야 한다.

---

## 4. 검색이 아니라 탐색

일반적인 검색창처럼 “정확한 제목을 입력하는 구조”는 도서관스럽지 않다.

사람은 실제 도서관에서 책 제목을 정확히 알고 검색하지 않는다.

대신 이런 식으로 접근한다.

```txt
AI 처음인데...
프롬프트 잘 쓰고 싶어
LLM이 어떻게 동작하지?
```

따라서 Dechive의 검색은 **검색창**이라기보다 **탐색창**이어야 한다.

### 탐색창 placeholder 예시

```txt
어떤 지식을 찾고 있나요?
LLM 처음인데...
프롬프트 잘 쓰고 싶어
AI 구조를 이해하고 싶어
```

---

## 5. 우선순위

기능 우선순위는 다음과 같다.

1. 탐색창/검색
2. 총 기록/시리즈 카운트 표시
3. 시리즈를 책 단위로 표현
4. 태그 기반 연관 글
5. 이전/다음 챕터 네비게이션
6. 무작위 기록 펼쳐보기
7. 사서 인터랙션 개선

---

## 6. Archive 페이지 방향

현재 Archive 페이지는 컨셉 자체는 맞다.

- 왼쪽: 분류/시리즈
- 오른쪽: 기록 목록
- 전체: 펼쳐진 책 UI

하지만 현재는 “읽는 구조”는 있으나 “탐색 구조”가 약하다.

### Archive에 추가하면 좋은 것

#### 1. 탐색창

상단 또는 책 내부 상단에 배치.

```txt
어떤 지식을 찾고 있나요?
```

#### 2. 총 기록 수

```txt
총 기록 26개 · 시리즈 4개
```

쌓여 있는 느낌을 빠르게 준다.

#### 3. 시리즈를 책처럼 표현

현재:

```txt
프롬프트 가이드 19편
```

개선:

```txt
프롬프트 가이드
19편 · 진행 중
```

또는

```txt
프롬프트 가이드
19개의 기록
```

단, 난이도/완주 보상 같은 표현은 Udemy/강의 플랫폼 느낌이 나므로 피하는 것이 좋다.

#### 4. 연관 기록

글 상세 페이지 또는 오른쪽 하단에 배치.

```txt
관련 기록
- Few-shot 학습
- Chain of Thought
- 프롬프트 구조 설계
```

#### 5. 이전/다음 기록

```txt
← 이전 기록 | 다음 기록 →
```

#### 6. 랜덤 탐색

```txt
아무 기록이나 펼쳐보기
우연히 발견하기
```

## 7. 메인 페이지 방향

메인은 정보를 많이 보여주는 곳이 아니다.

> 메인은 도서관 입구다.

사용자가 들어오자마자 해야 할 행동이 보여야 한다.

### 메인 중앙에 들어갈 것

중앙에는 딱 3개만 들어가는 것이 좋다.

```txt
지식을 탐색하는 도서관

[ 어떤 지식을 찾고 있나요? ]

[ 탐색 시작 ] [ 아무 기록이나 펼쳐보기 ]
```

또는 기존 카피를 유지할 경우:

```txt
생각이 기록이 되는 순간,
의미를 가진다.

[ 어떤 지식을 찾고 있나요? ]

[ 탐색 시작 ] [ 아무 기록이나 펼쳐보기 ]
```

### 메인에 넣으면 안 되는 것

- 긴 설명 문단
- 글 목록 전체
- 기능 설명 나열
- 너무 많은 버튼

메인은 “읽게 하는 곳”이 아니라 “들어가게 만드는 문”이다.

## 8. 가로로 긴 화면 대응

가로로 긴 화면에서는 중앙 콘텐츠만 있으면 좌우가 죽는다.

문제 구조:

```txt
[ 검은 여백 ] [ 중앙 콘텐츠 ] [ 검은 여백 ]
```

이 여백을 그냥 비워두지 말고 도서관의 확장 공간처럼 활용할 수 있다.

### 추천 구조

```txt
[ 좌측: 탐색/태그/카운트 ] [ 중앙: 카피 + 탐색창 ] [ 우측: 추천/랜덤/사서 ]
```

단, 너무 많은 UI를 올리면 분위기가 깨질 수 있으므로 메인에서는 최소화한다.

## 9. 모바일 배경 방향

모바일은 데스크탑 이미지를 억지로 자르면 안 된다.

모바일 전용 세로 배경을 따로 뽑는 것이 좋다.

### 모바일 배경 조건

```txt
상단: 어두운 도서관 천장 / 책장
중앙: 텍스트와 탐색창을 위한 빈 공간
하단: 음각 문양 또는 도서관 바닥
오른쪽 하단 또는 하단: 해골 사서/책/랜턴
```

파일은 분리 추천:

```txt
desktop-bg.webp
mobile-bg.webp
```

## 10. 마스코트의 역할

해골 마스코트는 단순 장식이 아니다.

> 이 해골은 Dechive의 사서다.

하지만 중요한 점:

> 배경 이미지 속 해골은 실제 사서 인터랙션이 아니다.

해상도와 화면비가 모두 다르기 때문에 배경 속 해골을 클릭 가능한 사서로 쓰면 위치가 깨질 수 있다.

따라서 역할을 분리해야 한다.

### 배경 해골

- 분위기
- 브랜드 감성
- 도서관 사서의 존재감

### UI 해골

- 실제 사서 버튼
- 채팅 인터페이스
- 추천/탐색 안내
- 화면 크기와 무관하게 고정 가능

추천 구조:

```txt
배경: 오른쪽 하단에 해골 사서가 앉아 있음
UI: 오른쪽 하단 fixed 버튼 또는 말풍선으로 사서 인터랙션 제공
```

예시:

```txt
사서 대기
궁금한 것이 있다면 물어보세요.
```

## 11. AI 사서 답변 방향

현재 사서 답변은 “온라인 강의 추천봇”처럼 느껴질 수 있다.

현재 방식:

```txt
LLM 처음이면 이 글부터 읽고, 다음은 이것, 다음은 이것...
```

이건 선형 커리큘럼 느낌이다.

도서관 사서처럼 보이려면 “정답 경로”보다 “갈림길”을 제공해야 한다.

### 개선된 답변 예시

```txt
LLM을 처음 접하신다면, 이 기록이 가장 편한 출발점이에요.

→ LLM 작동 원리와 프롬프트가 효과적인 이유

이후에는 관심에 따라 이렇게 둘러볼 수 있어요.

정확도를 높이고 싶다면
→ Few-shot 학습과 Chain of Thought

오류를 줄이고 싶다면
→ AI 할루시네이션 방지법

프롬프트 구조를 잡고 싶다면
→ AI 페르소나 설정법
```

핵심:

> 사서는 정답을 주는 사람이 아니라 길을 열어주는 사람이다.

## 12. 배경 이미지 방향

최종적으로 선택한 방향은 다음과 같다.

- 고딕 도서관
- 좌우 책장
- 중앙은 UI를 위한 어두운 빈 공간
- 오른쪽 하단에 해골 사서
- 바닥은 빛나는 마법진이 아니라 음각 문양
- 오로라/네온/강한 마법 효과 제거
- 촛불과 랜턴의 따뜻한 빛 유지

### 피해야 할 것

- 중앙이 너무 밝게 빛나는 것
- 컬러 오로라
- 강한 마법진
- 중앙에 오브젝트가 많아지는 것
- 해골이 너무 커지는 것
- 게임 로비처럼 보이는 것

### 좋은 상태

- 중앙 텍스트 가독성이 좋음
- 마스코트가 보이지만 과하지 않음
- 음각 문양은 질감처럼 존재
- 전체적으로 조용한 지식 도서관 느낌

## 13. 중앙 빛에 대한 판단

중앙은 “빛나면” 안 된다.

정확한 방향은 다음과 같다.

```txt
중앙을 밝게 만든다 ❌
중앙에 빛이 존재하는 느낌만 준다 ⭕
```

즉, 완전 검은 벽처럼 보이면 평면적이므로 아주 약한 공간감은 필요하다.

하지만 빛이 눈에 띄면 실패다.

성공 기준:

```txt
“오, 빛 예쁘다” → 실패
“뭔가 깊이가 있는데 정확히 뭐 때문인지는 모르겠다” → 성공
```

## 14. CSS 오로라 판단

기존 AuroraLight 코드는 중앙을 크게 밝히는 글로우에 가까웠다.

기존 코드 문제:

- 80vw/70vw/60vw로 너무 큼
- 중앙 전체를 덮음
- 오로라보다는 뿌연 빛 덩어리처럼 보임
- 배경과 레이어가 꼬일 수 있음

결론:

> 지금 최종 배경에서는 오로라를 빼는 것이 더 고급스럽다.

필요하면 이미지 자체보다 UI 검색창/버튼에 반투명 블랙 레이어를 쓰는 것이 더 안전하다.

---

## 15. 현재 추천 배경 상태

오로라를 뺀 순수 배경 100 상태가 가장 안정적이었다.

평가:

- 중앙 가독성 좋음
- 마스코트 존재감 좋음
- 도서관 느낌 진지함
- 강한 판타지 효과 없음
- UI 올리기 좋음

추천:

```txt
배경 밝기: 현재 100 유지
오로라: 제거
중앙 UI: 반투명 블랙 + 얇은 border
WebP: q 90
```

## 16. WebP 변환

메인 히어로 배경은 첫인상이 중요하므로 `cwebp` 품질은 90 추천.

```bash
cwebp -q 90 input.png -o main-bg.webp
```

용량이 너무 크면 85까지 낮춰볼 수 있다.

```bash
cwebp -q 85 input.png -o main-bg.webp
```

기준:

```txt
메인 히어로 배경: q 90
본문/카드 이미지: q 80~85
썸네일: q 75~80
```

## 17. 중앙 UI 예시

현재 배경 위에 탐색창을 올릴 경우, 배경을 더 밝히지 말고 UI 자체에 레이어를 주는 것이 좋다.

```jsx
<div className="mt-8 w-full max-w-xl rounded-2xl border border-white/15 bg-black/35 px-5 py-4 backdrop-blur-md">
  <input
    placeholder="어떤 지식을 찾고 있나요?"
    className="w-full bg-transparent text-white placeholder:text-white/45 outline-none"
  />
</div>
```

버튼 예시:

```jsx
<div className="mt-5 flex items-center justify-center gap-3">
  <button className="rounded-full border border-white/15 bg-white/10 px-5 py-3 text-sm text-white backdrop-blur-md hover:bg-white/15">
    탐색 시작
  </button>
  <button className="rounded-full border border-amber-200/20 bg-amber-100/10 px-5 py-3 text-sm text-amber-100 backdrop-blur-md hover:bg-amber-100/15">
    아무 기록이나 펼쳐보기
  </button>
</div>
```
